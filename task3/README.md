# Домашнее задание №4
## Сервис подписок и обновление архитектуры

---

## Контекст

На данный момент система умеет получать информацию о репозиториях GitHub по запросу пользователя. В этом задании архитектура эволюционирует: пользователь может **подписаться** на интересующие его репозитории, а система будет знать, какие репозитории отслеживать.

---

## Задача

Необходимо реализовать функциональность сервиса **Subscribe** и доработать существующие сервисы **Collector** и **Processor**.

---

## Что нужно реализовать

### 1. Сервис Subscribe 

Сервис управляет подписками пользователей на репозитории. Он должен:

- Хранить список подписок в базе данных **PostgreSQL**
  - Используются миграции для инициализации базы данных
- Предоставлять **gRPC-интерфейс** для работы с подписками --- необходимо расиширить\переработать существующий шаблон
- Самостоятельно ходить в **GitHub REST API**, чтобы проверять существование репозитория при добавлении подписки

### 2. Обновление сервиса API

Сервис API расширяется новыми эндпоинтами для управления подписками. **Существующее API из предыдущих заданий продолжает работать без изменений.**

Новые эндпоинты:

- `POST /subscriptions` — подписаться на репозиторий
- `DELETE /subscriptions/{owner}/{repo}` — отписаться
- `GET /subscriptions` — получить список подписок
- `GET /subscriptions/info`

Первые три эндпоинта ходят в **Subscribe по gRPC**.

### 3. Обновление сервиса Collector

Collector получает новую логику запроса данных. При вызове `GET /subscriptions/info` (через цепочку API → Processor → Collector) Collector:

1. Сам запрашивает у **Subscribe** актуальный список подписок по gRPC
2. Собирает информацию по каждому репозиторию из этого списка
3. Возвращает агрегированный результат в Processor

> Старый флоу (когда пользователь явно передаёт ссылку на репозиторий) остаётся рабочим и не ломается.

### 4. Доработка сервиса Processor

Processor меняется незначительно --- добавьте необходимую логику запроса репозиториев, на которых есть подписка.

---

## Proto-контракты

В рамках этого задания необходимо спроектировать и реализовать один контракт: **Collector → Subscribe** --- Контракт для получения Collector-ом актуального списка подписок перед тем, как идти в GitHub.

Также не забудьте внести необходимые изменения в существующие контракты API → Subscribe и Collector → Processor

---

## Требования

- Для работы с PostgreSQL использовать [**sqlc**](https://github.com/sqlc-dev/sqlc) для генерации типобезопасного кода из SQL-запросов
- Использовать [**golang-migrate**](https://github.com/golang-migrate/migrate) для работы с миграциями
- Для подключения к БД использовать [**pgx**](https://github.com/jackc/pgx) в режиме пула соединений (`pgxpool`)
- Конфигурация (DSN базы, адреса сервисов) — через конфиг файлы
- Обработка ошибок: репозиторий не существует на GitHub, дублирующая подписка, недоступность БД
- Каждый сервис запускается отдельно (отдельный бинарник или директория `cmd/`)
- В README описать, как поднять все сервисы локально (можно с `docker-compose`)
