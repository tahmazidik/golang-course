# Домашнее задание №5
## Асинхронный сбор данных, Kafka и кэш в Processor

---

## Контекст

На предыдущем этапе система уже умеет:

- получать информацию о репозиториях GitHub,
- хранить подписки пользователей,
- агрегировать данные по подпискам.

Теперь архитектура должна эволюционировать дальше: сервис **Processor** становится центральной точкой доступа к данным о репозиториях и начинает работать как кэш и как фоновый оркестратор обновлений.

Ключевая идея нового этапа:

- **Processor** больше **не вызывает Collector напрямую**,
- связь между **Processor** и **Collector** строится **через Kafka**,
- **Processor** хранит собранную информацию о репозиториях в собственной **PostgreSQL** базе,
- при запросе **Processor сначала пытается взять данные из своей БД**,
- если данных нет, **Processor ставит задачу в Kafka**,
- **Collector** забирает задачу из Kafka, идет в GitHub API и публикует результат обратно в Kafka,
- **Processor** читает результат из Kafka, сохраняет его в БД и затем может отдавать его по запросу,
- дополнительно **Collector раз в 15 секунд запрашивает список подписок у Subscribe** и инициирует фоновое обновление данных по ним через Kafka.

---

## Задача

Необходимо доработать систему так, чтобы она работала по асинхронной схеме сбора и хранения данных.

В итоговой архитектуре участвуют следующие сервисы:

1. **Processor**
2. **Collector**
3. **Subscribe**

---

## Что нужно реализовать

### 1. Обновление сервиса Processor

Processor становится основным сервисом для работы с данными о репозиториях.

Он должен:
 
- быть **producer/consumer** для Kafka,
- хранить информацию о репозиториях в **PostgreSQL**,
- по запросу:
    - сначала искать данные в своей БД,
    - если данные есть — возвращать их из БД,
    - если данных нет — публиковать задачу на сбор в Kafka.

Кроме того, Processor должен:

- читать сообщения из Kafka с результатами сбора данных,
- сохранять или обновлять записи в своей БД.

---

### 2. Обновление сервиса Collector

Collector больше не принимает запросы по gRPC.

Теперь Collector должен:

- читать задачи на сбор информации о репозиториях из Kafka,
- по каждой задаче обращаться в GitHub REST API,
- формировать результат,
- публиковать результат обратно в Kafka.

Collector остается сервисом, который инкапсулирует всю логику работы с GitHub API.

---

### 3. Фоновое обновление подписок

Раз в **15 секунд** Collector должен:

- вызывать Subscribe по gRPC,
- получать список всех подписок,
- для каждой подписки отправлять задачу на обновление в Kafka.

Это нужно для того, чтобы данные по подпискам в Processor периодически актуализировались без прямого запроса пользователя.

---

## Требования
    
- Взаимодействие между **Processor** и **Collector** — строго **через Kafka**.
- В каждом сервисе должно сохраняться разделение на слои:
    - **Domain**
    - **Use Cases**
    - **Controller/Handler**
    - **Adapter/Repository**
- **Processor** должен использовать **PostgreSQL** для хранения информации о репозиториях.
- Актуальное описание Swagger для api gateway 
- Для работы с PostgreSQL использовать:
    - [**sqlc**](https://docs.sqlc.dev/en/latest/tutorials/getting-started-postgresql.html)
    - [**golang-migrate**](https://github.com/golang-migrate/migrate)
    - [**pgxpool**](https://pkg.go.dev/github.com/jackc/pgx/v5/pgxpool)
- Конфигурация сервисов должна быть вынесена в конфиг-файлы.
- Для всех сервисов должны быть подготовлены **Dockerfile**.
- В `docker-compose.yaml` должны быть добавлены:
    - Kafka,
    - PostgreSQL для Processor,
    - все необходимые сервисы приложения.
- Ошибки должны обрабатываться корректно:
    - некорректный URL репозитория,
    - репозиторий не существует,
    - ошибка GitHub API,
    - недоступность Kafka,
    - недоступность БД Processor,
    - недоступность БД Subscribe,
    - ошибки сериализации/десериализации сообщений.

## Формат сдачи

Аналогично предыдущим заданиям:

- работа ведется в отдельной ветке,
- PR открывается в `main` вашего форка,
- в PR должны быть отмечены ревьюеры:
    - https://github.com/suvorovrain
    - https://github.com/Dabzelos
    - https://github.com/vacmannnn

В PR необходимо приложить:

1. скриншот запроса, который инициирует сбор данных,
2. скриншот повторного запроса, который возвращает данные из Processor,
3. краткое описание топиков Kafka и формата сообщений.

Задание засчитывается, если:

- проходят тесты CI,
- сервисы поднимаются локально,
- Kafka действительно используется как единственный канал связи между Processor и Collector,
- Processor действительно читает данные из своей БД, а не ходит в Collector напрямую.
